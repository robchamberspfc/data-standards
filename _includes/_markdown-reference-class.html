Reference

This page refers to HACT Model Version Main/3.5 published on 16 Jun 2022 00:00

Technical users who are really to apply this standard may choose to go straight to the technical guidance

Summary

Reference is one of the entities in the HACT data standard.

This entity records an identifier and where it came from. It sometimes gives other details about the identifier.

An identifier is a code, usually a string of numbers or letters, that is assigned to a real-world thing, such as a building or a tenant. It can then be used to identify that thing.

Without using the reference entity to show an identifier’s source, there are several ways it could become less reliable. For example, two things might have the same identifier, or one thing might have been given several different identifiers. Knowing the source of each identifier makes it possible to resolve these problems and provides a single version of the truth.

A reference is a bit like the:

  • the details on a credit card, where the name of the bank shows who assigned the long number that uniquely identifies that specific card
  • an email address, where the @example.com part shows which domain the firstname.lastname identifier is associated with
  • a telephone number, where country code shows which country the rest of the number is from

Reference is metadata, stored outside of the main table.

About the attributes of this entity

Reference has attributes that show:

  • the reference number (or other identifier) in question
    • the organisation or system that created the reference number
    • what role the source organisation has, such as property manager or facility manager
    • any important background about the reference, for example that it is a barcode for something physical

See the technical guidance for the attributes

Who reference is for

The reference entity is useful for teams who:

  • manage data and make sure it is accurate
  • use data and need it to be reliable

Accurate and reliable data can improve the experience of tenants in many ways.

It can also improve things for internal teams, including:

  • data professionals
  • housing staff
  • housing officers
  • management

How it benefits your organisation

Housing providers’ data can become messy. For example, you might have multiple different identifiers in your system for a building. One might be from an external source, while two others are internal and system-generated.

Using the reference entity can help cleanse these sorts of messes, by making it possible to:

  • remove confusion
    • spot duplicates
    • capture new data in the right place
    • distinguish between identifiers
    • see whether an identifier is official
    • share data between systems and organisations, or through APIs
